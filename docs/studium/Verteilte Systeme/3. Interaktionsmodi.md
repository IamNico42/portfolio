## Interaktionsmodi

Die Grundfrage ist:

> Wie bekommt ein Client mit, dass auf dem Server etwas Neues passiert ist?

Beispiel: Chat-App, Statusanzeige, Live-Ticker, Benachrichtigung, neue Bestellung, Spielerpositionen, neue Nachricht.

## Problem: Der Server weiß etwas Neues, aber der Client noch nicht

```
Client zeigt Chat an
Server bekommt neue Nachricht
Client muss irgendwie davon erfahren
```

Jetzt gibt es mehrere Möglichkeiten

## 1. Polling

> Der Client fragt den Server regelmäßig, ob es etwas Neues gibt.

Beispiel:
```
Client: Gibt es neue Nachrichten?
Server: Nein.

5 Sekunden später:

Client: Gibt es neue Nachrichten?
Server: Nein.

5 Sekunden später:

Client: Gibt es neue Nachrichten?
Server: Ja, hier ist eine neue Nachricht.
```

Diagramm:
```
Client                Server
  | --- Anfrage -----> |
  | <--- Nein -------- |
  |                    |
  | --- Anfrage -----> |
  | <--- Nein -------- |
  |                    |
  | --- Anfrage -----> |
  | <--- Nachricht --- |
```

Einfach zu bauen, aber extrem ineffizient weil viele Anfragen einfach leer sind.

>Beim Polling fragt der Client den Server in regelmäßigen Abständen nach neuen Informationen. Der Vorteil ist die einfache Umsetzung, der Nachteil ist unnötiger Kommunikationsaufwand durch viele Anfragen ohne neue Daten.

## 2. Long Polling


```
Gibt es etwas Neues?
```

Aber der Server antwortet nicht sofort mit „Nein“.

Stattdessen hält der Server die Anfrage offen, bis:
1. etwas Neues da ist  
    oder
2. ein Timeout erreicht wird

Beispiel:
```
Client: Gibt es neue Nachrichten?
Server wartet...
Server wartet...
Neue Nachricht kommt rein.
Server: Ja, hier ist die Nachricht.
Client stellt direkt neue Long-Polling-Anfrage.
```

Diagramm:
```
Client                         Server
  | -------- Anfrage --------> |
  |                            | wartet...
  |                            | wartet...
  | <------ Nachricht -------- |
  |                            |
  | -------- neue Anfrage ---> |
```

#### Vorteil
- Weniger unnötige Antworten als normales Polling.
- Der Server sagt nicht ständig „Nein“, sondern wartet, bis etwas passiert.
#### Nachteil
- Der Server muss viele offene Verbindungen halten.
- Wenn 10.000 Clients long-pollen, dann gibt es 10.000 offene wartende Requests. Das kann Ressourcen kosten.

>Beim Long Polling sendet der Client eine Anfrage, die vom Server so lange offen gehalten wird, bis neue Daten verfügbar sind oder ein Timeout eintritt. Dadurch werden unnötige leere Antworten reduziert, allerdings muss der Server viele offene Anfragen verwalten.


## 3. Push

> Der Server sendet aktiv Daten an den Client, sobald etwas Neues passiert.

Der Client muss nicht ständig fragen.
Beispiel Chat:
```
Neue Nachricht kommt beim Server an.
Server pusht Nachricht direkt an alle betroffenen Clients.
```

Diagramm:
```
Client                         Server
  | <------ Nachricht -------- |
  | <------ Nachricht -------- |
  | <------ Nachricht -------- |
```

Typische Technik dafür:

```
WebSocket
Server-Sent Events
Push Notifications
```

#### Vorteil
- Sehr effizient für echte Echtzeit-Kommunikation.
- Neue Daten kommen sofort an.
#### Nachteil
- Komplexer als normales Request-Response.
- Man braucht meistens eine dauerhaft offene Verbindung oder ein Benachrichtigungssystem.

> Beim Push-Modell sendet der Server neue Informationen aktiv an den Client, sobald sie verfügbar sind. Der Client muss nicht regelmäßig nachfragen. Das ist effizient für Echtzeit-Anwendungen, erfordert aber meist komplexere Infrastruktur oder dauerhafte Verbindungen.

## Zusammenfassung

| Modus        | Wer startet?           | Wie funktioniert es?                    | Gut für                                |
| ------------ | ---------------------- | --------------------------------------- | -------------------------------------- |
| Polling      | Client                 | Client fragt regelmäßig                 | einfache Statusabfragen                |
| Long Polling | Client                 | Client fragt, Server wartet mit Antwort | quasi Echtzeit ohne echtes Push        |
| Push         | Server nach Verbindung | Server sendet aktiv neue Daten          | Chat, Live-Updates, Benachrichtigungen |

>Polling, Long Polling und Push sind Interaktionsmodi, mit denen ein Client über neue Informationen informiert werden kann. Beim Polling fragt der Client den Server regelmäßig nach neuen Daten. Das ist einfach, erzeugt aber viele unnötige Anfragen. Beim Long Polling stellt der Client ebenfalls eine Anfrage, der Server beantwortet diese aber erst, wenn neue Daten verfügbar sind oder ein Timeout eintritt. Dadurch werden leere Antworten reduziert. Beim Push-Modell sendet der Server neue Informationen aktiv an den Client, sobald sie entstehen. Das eignet sich besonders für Echtzeit-Anwendungen, ist aber technisch aufwendiger.
