## 1. Was bedeutet Nebenläufigkeit?

**Nebenläufigkeit** bedeutet:

> Mehrere Abläufe überschneiden sich zeitlich.

Beispiel:
```
Client A sendet Anfrage
Client B sendet Anfrage
Client C sendet Anfrage
Server bearbeitet mehrere Dinge gleichzeitig oder zeitlich überlappend
Es wirkt gleichzeitig, tatsächlich wechselt die CPU schnell zwischen den Abläufen.
```

In verteilten Systemen ist das normal, weil viele Clients, Dienste und Komponenten gleichzeitig aktiv sein können.

Einfaches Beispiel
```
100 Nutzer öffnen gleichzeitig deine Website.
```

Dann will man nicht
```
Nutzer 1 wird komplett fertig bearbeitetdanach Nutzer 2danach Nutzer 3...
```

Das wäre extrem langsam.
Man möchte:
```
mehrere Anfragen gleichzeitig / überlappend bearbeiten
```


## 2. Was bedeutet Parallelität

**Parallelität** bedeutet:
> Mehrere Abläufe werden tatsächlich gleichzeitig ausgeführt.

Zum Beispiel auf mehreren CPU-Kernen oder mehreren Rechnern:

```
CPU-Kern 1: Task A
CPU-Kern 2: Task B
```

Merksatz:
> Nebenläufig = zeitlich überlappend.  
> Parallel = tatsächlich gleichzeitig.


## 3. Warum brauchen wir Nebenläufigkeit und Parallelität?

Weil sonst Anfragen streng nacheinander verarbeitet würden. Das wäre bei vielen Nutzern viel zu langsam.

Mit Nebenläufigkeit:
```
Client A ─┐
Client B ─┼→ Server bearbeitet mehrere Anfragen überlappend
Client C ─┘
```

Vorteile:
- bessere Ressourcenauslastung
- mehr Durchsatz
- geringere Wartezeiten
- mehrere Nutzer gleichzeitig bedienbar
- Verteilung auf mehrere Rechner nutzbar


## 4. Problem: Geteilte Ressourcen

Nebenläufigkeit wird problematisch, wenn mehrere Abläufe gleichzeitig oder überlappend auf dieselbe Ressource zugreifen.

**Race Condition**
Das Ergebnis hängt von der zufälligen zeitlichen Reihenfolge der Abläufe ab.
```
A liest Kontostand 100 €
B liest Kontostand 100 €
A schreibt 150 €
B schreibt 70 €
```

**Deadlock**
Mehrere Abläufe warten dauerhaft gegenseitig aufeinander.
```
A besitzt Ressource 1 und wartet auf Ressource 2
B besitzt Ressource 2 und wartet auf Ressource 1
```

**Livelock**
Die Abläufe sind aktiv und reagieren ständig, machen aber keinen Fortschritt.
```
A weicht B aus
B weicht gleichzeitig A aus
beide wiederholen das ständig
```

**Starvation**
Ein Ablauf kommt dauerhaft oder sehr lange nicht zum Zug, während andere weiterarbeiten.
Wenn der Scheduler gewissen Prozessen eine niedrige Priorität zuweist und nicht zum Zug kommen.

## 5. Wie beheben wir diese Probleme

**Kritischer Bereich**
Ist der Bereich in einem Code der als kritisch betrachtet wird - zB Zugriff auf Ressourcen.

### **Mutual Exclusion**
Mutual Exclusion bedeutet, dass höchstens ein Ablauf gleichzeitig den kritischen Bereich betreten darf.
Beispielsweise mit **Locks** - verhindert Race Conditions:
```
A erhält Lock
A bearbeitet Ressource

B möchte Lock
B muss warten

A gibt Lock frei
B erhält Lock
```

#### Koordination bei verteilten Systemen

**Zentraler Koordinator**
Eine zentrale Instanz entscheidet, wer den kritischen Bereich betreten darf.
```
REQUEST → REPLY → Zugriff → RELEASE

Vorteile
- einfache Umsetzung
- klare Reihenfolge
- geringer Kommunikationsaufwand

Nachteile
- Flaschenhals
- Single Point of Failure
```

**Verteilter gegenseitiger Ausschluss**
Die Komponenten stimmen sich untereinander ab.
```
- Lamport
- Ricart/Agrawala
- Token-Verfahren

Vorteile
- kein zentraler Koordinator
- Last verteilt

Nachteile
- mehr Kommunikationsaufwand
- komplexere Koordination
```

**Token-Verfahren (Ticket System)**
Nur die Komponente mit dem Token darf den kritischen Bereich betreten.
```
Vorteile
- eindeutige Zugriffsberechtigung

Nachteile
- Token kann verloren gehen
- Token-Besitzer kann ausfallen
- Wartezeiten möglich
```
