## 1. Was ist überhaupt ein System?
Ein **System** ist erstmal nur eine Menge von Teilen, die zusammenarbeiten, um eine Aufgabe zu erfüllen.

Beispiel einfache Website:

Browser ->Webserver -> Datenbank
Minecraft Client -> Game-Server -> Datenbank


Das Ganze ist ein System, weil diese Teile zusammen eine Aufgabe erfüllen: Nutzer ruft Seite auf, Server verarbeitet Anfrage, Datenbank liefert Daten, Browser zeigt Ergebnis.

Jetzt kommt die zentrale Frage im Kurs:

> Ist dieses System zentralisiert oder verteilt?

## 2. Zentralisiertes System vs. verteiltes System
### **zentralisiertes System**
Ein **zentralisiertes System** läuft im Kern an einer Stelle. Alles Wichtige passiert auf einem Rechner oder in einem Prozess.

Beispiel:
Eine kleine lokale Notizen-App auf deinem Laptop. Die Daten liegen lokal, die Logik läuft lokal, kein anderer Rechner ist beteiligt.


### **Verteiltes System**
Ein verteiltes System besteht dagegen aus mehreren eigenständigen Komponenten, die auf mehreren Rechnersystemen laufen.

>[!quote] Nebenbemerkung
>"Ein verteiltes System besteht aus mindestens zwei Software-Komponenten, die auf mindestens
>zwei verschiedenen Rechnersystemen ablaufen. Außerdem ist es genau dann ein verteiltes System, wenn diese Komponenten kommunizieren. Eine Komponente ist ein eigenständig ablauffähiges Programm; eine Bibliothek oder ein Framework ist keine Komponente."

Es reicht nicht wenn z. B. mehrere User auf einem System diverse Programme laufen lassen die miteinander kommunizieren, sondern es sind zwingend zwei Rechnersysteme notwendig - Das kann allerdings auch z. B. Client -> Server beinhalten - sobald diese beiden Komponente miteinander kommunizieren können wir von einem verteilten System sprechen.

## 3. Warum überhaupt verteilte Systeme?
Man verteilt Systeme weil oft ein einzelner Rechner, einzelnes Programm oder ein einzelner Ort irgendwann Grenzen hat.
Das können lokale Schranken, Netzausbau, eine stärkere verteilte Welt und technologischen Fortschritt sein.

### 3.1 Verteilte Aufgabenstellung
Manche Probleme sind von Natur aus verteilt.
Beispiel: Eine Mobilitätsplattform muss Daten von ÖPNV, Sharing-Anbietern, Buchungssystemen und Zahlungsdiensten integrieren. Diese Dinge gehören nicht alle einem einzigen System und liegen nicht alle auf einem Rechner.
Oder: Ein Webshop, der PayPal nutzt. Der Shop läuft bei Anbieter A und PayPal läuft bei Anbieter B - trotzdem müssen beide zusammenarbeiten.

### 3.2 Robustheit, Zuverlässigkeit, Verfügbarkeit
Wenn alles auf einem Rechner läuft und dieser ausfällt, dann läuft gar nichts mehr. Bei Verteilten Systemen kann man Dienste mehrfach betreiben. Falls ein Server ausfällt, kann ein anderer übernehmen.

### 3.3 Spezial-Hardware nutzen
Manchmal hat ein anderer Rechner besondere Hardware.

Beispiel:
Ein KI-Service nutzt GPUs. Die normale Anwendung hat keine GPU, ruft aber einen Dienst auf, der auf GPU-Server läuft.

### 3.4 Freie Ressourcen ausnutzen
Wenn viele Rechner im Netzwerk sind kann man Arbeit auf mehrere Maschinen verteilen.

### 3.5 Teilen von Ressourcen

Distribution means Sharing. "Verteilen" bedeutet auch, dass etwas Begrenztes geteilt wird. Eine Ressource kann physisch, virtuell oder funktional sein, zum Beispiel CPU-Zeit, Speicher, Dateien, Datenbanken, Datensätze, Informationen oder Funktionen. Ein Dienst verwaltet solche Ressourcen und stellt sie anderen, auch entfernten Komponenten über eine Schnittstelle zur Verfügung.

>[!tip] Merke
>Ressource = etwas Begrenztes, das nicht jeder selbst hat.
>Dienst = macht diese Ressource für andere nutzbar.
>Verteilung = andere Komponenten können über Kommunikation darauf zugreifen.


## 4. Was sind die Nachteile verteilter Systeme?
Kommunikationskosten, Komplexität bei Entwicklung und Komplexität im Betrieb sind grundlegende Probleme bzw. Nachteile.

### 4.1 Kommunikationskosten
Wenn zwei Komponenten auf demselben Rechner sind, können sie direkt Speicher/Funktionen benutzen - bei verschiedenen Rechnern läuft es über das Netzwerk.

Das kostet:
- Zeit
- Bandbreite
- Protokollaufwand
- Fehlerbehandlung

Beispiel:
Ein lokaler Funktionsaufruf ist extrem schnell. Ein HTTP-Request an einen Server kann Millisekunden oder Sekunden dauern und kann fehlschlagen.

### 4.2 Komplexität bei Entwicklung
In einem zentralen Programm ist vieles einfacher:
Man ruft Funktionen auf und bekommt Ergebnisse zurück.

In einem verteilten System kann viel mehr schief gehen:
- Was passiert wenn der andere Dienst nicht antwortet
- Was passiert wenn die Nachricht verloren geht
- Was passiert wenn zwei Nutzer gleichzeitig dieselbe Ressource ändern (Race condition)
- Was passiert wenn ein Teil alt und ein anderer Teil neu deployed ist?

### 4.3 Komplexität im Betrieb
Man muss mehrere Komponenten überwachen:
Server, Datenbanken, Netzwerke, APIs, Logos, Versionen, Deployments, Sicherheitsregeln.

Fehler sind ebenfalls schwerer zu finden weil es einen viel längeren Weg gibt zwischen Request und Response.


## 5. Ressource: Das erste echte Grundkonzept
Eine **Ressource** ist etwas Begrenztes, auf das zugegriffen wird.

Die Folien definieren Ressource als ein physikalisches oder virtuelles Element eines Rechnersystems, das nur begrenzt zur Verfügung steht. Beispiele sind CPU, GPU, Speicher, Festplatten, Geräte, Prozesse, Threads, Dateien, Datenbanken, Daten und Funktionen.

### Einfach gesagt
Eine Ressource ist etwas, das jemand benutzen will, aber das nicht unbegrenzt verfügbar ist.

Beispiele:

- Eine Datei
- Ein Datenbankeintrag
- Ein Drucker
- CPU-Zeit
- Speicherplatz
- Eine Funktion wie „Zahlung ausführen“
- Ein Datensatz wie „User mit ID 42“

### Warum ist das in verteilten Systemen wichtig?
Weil mehrere Komponenten auf dieselbe Ressource zugreifen können.

Beispiel:  
Zwei Nutzer bearbeiten gleichzeitig denselben Account.  
Oder zwei Services wollen denselben Datenbankeintrag ändern.

Dann entstehen Fragen:  
Wer darf wann zugreifen?  
Was passiert bei gleichzeitigem Zugriff?  
Wie bleibt der Zustand konsistent?

Das führt später zu Themen wie Nebenläufigkeit, Sperren, Koordination und Konsistenz.

## 6. Dienst / Service
Ein Dienst ist eine Anwendung oder Komponente, die Ressourcen verwaltet und anderen Prozessen bereitstellt.

Die Folie sagen: Ein Dienst ist eine Applikation auf einem Rechner, verwaltet verwandte Ressourcen und stellt diese Ressourcen entfernten Prozessen zur Verfügung. Zugriff erfolgt über eine Dienstschnittstelle, also einen Contract mit exportierten Operationen. Beispiele sind Dateidienst/NFS, Druckdienst, PayPal/giroPay, Web Services und REST-basierte Microservices.

### Einfach gesagt
Ein Dienst ist jemand, der etwas für andere anbietet.

Beispiel:  
PayPal bietet Zahlungsfunktionen an.  
Ein Dateiserver bietet Dateien an.  
Eine REST-API bietet Daten oder Funktionen an.  
Eine Datenbank bietet Zugriff auf gespeicherte Daten an.

### Wichtige Unterscheidung
Die **Ressource** ist das, was begrenzt ist oder verwaltet wird.
Der **Dienst** ist die Komponente, die den Zugriff darauf ermöglicht.

Beispiel:  
Datei = Ressource  
Dateiserver = Dienst  
`read()`, `write()`, `delete()` = Operationen der Dienstschnittstelle

## 7. Anbieter und Konsument

Eine Applikation, die Dienste anbietet, ist ein **Dienst-Anbieter**, also Service Provider beziehungsweise Server.  
Eine Applikation, die Dienste nutzt, ist ein **Dienst-Konsument**, also Service Consumer beziehungsweise Client.


**Anbieter** = stellt etwas bereit
**Konsument** = nutzt etwas

Beispiel:
Paypal = Anbieter eines Zahlungsdienstes
Webshop = Konsument dieses Zahlungsdienstes

Aber: Eine Komponente kann **gleichzeitig** Anbieter und Konsument sein.

Ein Backend bietet dem Browser eine API an. Gleichzeitig nutzt das Backend eine Datenbank oder PayPal.
Also:  
Browser → Backend → PayPal/Datenbank

Für den Browser ist das Backend ein Anbieter.  
Für PayPal ist das Backend ein Konsument.

## 8. Client und Server

Server wartet auf Clients und handled
Client initiiert Verbindung


[!quote] Prüfungsformulierung
> Im Client-Server-Modell bietet der Server einen Dienst an und wartet auf Anfragen. Der Client nutzt diesen Dienst, indem er die Kommunikation initiiert und Anfragen an den Server sendet.


## 9. Kommunikation
Ein verteiltes System existiert nicht nur deshalb, weil mehrere Programme irgendwo laufen. Sie müssen miteinander kommunizieren.

Eine Menge verteilter Software-Komponenten ist genau dann ein verteiltes System, wenn die Komponenten kommunizieren.

Eine Komponente sendet Informationen an eine andere Komponente.
Typisch:
- Anfrage
- Antwort
- Nachricht
- Event
- Datenpaket
- API-Call

## 10. Interaktion, Kommunikation, Kooperation

### Kommunikation

Reiner Informationsaustausch.

Beispiel:  
Client sendet Anfrage an Server.

### Interaktion

Kommunikation mit Bezug aufeinander. Eine Seite reagiert auf die andere.

Beispiel:  
Client fragt, Server antwortet.

### Kooperation

Mehrere Komponenten arbeiten gemeinsam an einem Ziel.

Beispiel:  
Webshop, Zahlungsdienst, Lagerdienst und Versanddienst arbeiten zusammen, damit eine Bestellung abgeschlossen wird.

>[!tip]Merksatz:
>Kommunikation = reden. 
>Interaktion = aufeinander reagieren. 
>Kooperation = gemeinsam Aufgabe lösen.


## Synchron und Asynchron

### Synchron:

Der Sender wartet auf Antwort bevor er weitermacht.

Beispiel:
Man klickt auf Login
Die App wartet bis der Server sagt: Login okay oder Passwort falsch

> [!tip]
> Synchron ist gut, wenn man sofort ein Ergebnis braucht.

Nachteil: Wenn der Server langsam ist dann wartet man.

### Asynchron

Der Sender schickt etwas los und macht erstmal weiter. Die Antwort kommt dann sobald der Server diese verarbeitet hat, aber die App ist bis dahin nicht eingeschränkt sondern kann weiter verwendet werden.

Beispiel:
Man lädt ein Video hoch. Die Plattform sagt "Upload erhalte"

> [!tip]
> Asynchron ist gut, wenn Wartezeiten sonst blockieren würden.

## 12. Polling, Long Polling, Push

### Polling
Client fragt regelmäßig:  
„Gibt es was Neues?“  
„Gibt es was Neues?“  
„Gibt es was Neues?“

Problem:  
Viele unnötige Anfragen.

### Long Polling
Client fragt:  
„Gibt es was Neues?“

Server antwortet nicht sofort, sondern wartet, bis etwas Neues da ist oder ein Timeout kommt.

Besser als Polling, aber immer noch Anfrage-basiert.

### Push
Server sendet aktiv an Client, sobald etwas passiert.

Beispiel:  
Chat-Nachricht kommt rein, Server pusht sie an den Client.

## 13. Transparenz

Der Nutzer soll nicht merken müssen, wo genau etwas liegt oder welcher Server gerade antwortet.

Beispiel:  
Du öffnest eine Website. Dir ist egal, ob die Daten aus Frankfurt, Irland oder den USA kommen. Es wirkt wie „eine Website“.

Tanenbaums Definition aus den Folien passt dazu:
> Eine Sammlung unabhängiger Computer erscheint den Nutzern als ein einzelnes kohärentes System.

[!quote] Wichtig:
> Verteilungstransparenz bedeutet, dass die Verteilung von Komponenten, Ressourcen oder Fehlern möglichst verborgen wird, sodass das System nach außen wie ein einheitliches System wirkt.


## 14. Qualitätsmerkmal und Entwurfskriterium

### 14.1 Skalierbarkeit

Es gibt verschiedene Dimensionen:

- Größe: mehr Nutzer, mehr Komponenten
- Geographie: mehr Standorte, weitere Entfernungen
- Administration: mehrere Organisationen/Verwaltungsbereiche

Ein Problem bei geographischer Skalierung ist zum Beispiel Latenz. Wenn Nachrichten über mehr Zwischenknoten müssen, können Antwortzeiten steigen, Knoten können Flaschenhälse bilden oder blockieren.

#### Einfach gesagt

Ein System skaliert gut, wenn es bei mehr Last, mehr Nutzern oder mehr Standorten nicht direkt kaputtgeht oder komplett neu gebaut werden muss.

### 14.2 Nebenläufigkeit

Nebenläufigkeit heißt:  
Mehrere Dinge passieren gleichzeitig oder überlappend.

Beispiel:  
Viele Nutzer schicken gleichzeitig Anfragen an denselben Server.

Problem:  
Wenn alle auf dieselbe Ressource zugreifen, können Konflikte entstehen.

Beispiel:  
Zwei Nutzer ändern gleichzeitig denselben Datensatz.

Dann braucht man Koordination:

- Sperren
- Transaktionen
- Reihenfolge
- Konfliktlösung

**Verteilte Systeme sind fast immer nebenläufig, weil mehrere Komponenten und Nutzer gleichzeitig aktiv sind.**

### 14.3 Kommunikationsgrad

Der Kommunikationsgrad beschreibt grob, wie eng oder technisch direkt Komponenten miteinander kommunizieren.

Je direkter und technischer die Kommunikation, desto mehr muss man selbst kontrollieren.

Beispiel:  
Socket = sehr niedrig/technisch  
REST-API = höher/abstrakter  
RPC/RMI = wirkt fast wie Funktions-/Methodenaufruf über Netzwerk


## 15 Zusammenfassung der Fundamente

In einem **zentralen System** ist vieles direkt:  
Funktionsaufrufe, Speicherzugriff, lokale Kontrolle, einfacher Zustand.

In einem **verteilten System** ist vieles indirekt:  
Nachrichten, Netzwerk, Latenz, Ausfälle, getrennte Zustände, parallele Zugriffe, Sicherheitsgrenzen, unterschiedliche Verantwortlichkeiten.


> [!tip] **8 - Merksätze**
> 1. Verteilt = Mehrere Programme + mehrere Rechner + Kommunikation
> 2. Eine Komponente ist eigenständig lauffähig; Library ist keine Komponente
> 3. Kommunikation ist Pflicht, sonst ist es kein verteiltes System
> 4. Ressource = begrenztes Ding, auf das zugegriffen wird.
> 5. Dienst = verwaltet Ressourcen und stellt sie anderen bereit.
> 6. Anbieter/Server bietet an; Konsument/Client nutzt
> 7. Verteilung bringt Vorteile, aber auch Kommunikationskosten und Komplexität+
