## 1. Was ist SCRUM?

Scrum ist ein **agiles Vorgehensmodell** für die Entwicklung komplexer Produkte. 
Die Arbeit wird nicht vollständig für Monate im Voraus geplant, sondern in kurzen Zeitabschnitten durchgeführt. Diese Abschnitte heißen **Sprints**.

```
Anforderungen auswählen
→ Sprint planen
→ entwickeln
→ Ergebnis prüfen
→ Feedback berücksichtigen
→ nächster Sprint
```

Scrum basiert stark auf drei Prinzipien:
- Transparenz
- Inspektion
- Anpassung

Das Team macht Arbeit und Fortschritt sichtbar, überprüft regelmäßig das Ergebnis und passt das weitere Vorgehen an.

## 2. SCRUM 3x3

SCRUM 3x3 beschreibt das System mit 3 Rollen, 3 Events und 3 Artefakten.

### 3 Rollen
#### Product Owner – PO

Der Product Owner ist für den **fachlichen Wert und die Priorisierung** verantwortlich.
- Anforderungen festlegen und erklären
- Product Backlog pflegen und priorisieren
- entscheiden, was fachlich besonders wichtig ist
- Fragen der Developer beantworten
- fertige User Stories prüfen und abnehmen
- Stakeholderinteressen vertreten

#### Scrum Master – SM

Der Scrum Master unterstützt den Scrum-Prozess.
- darauf achten, dass Scrum verstanden und eingehalten wird
- Meetings strukturieren und moderieren
- Kommunikationsprobleme beseitigen
- Hindernisse sichtbar machen
- Team bei der Selbstorganisation unterstützen
- nicht als klassischer Chef auftreten

Er unterstützt das Team dabei, selbst sinnvoll zu arbeiten.

#### Developer – DEV

Die Developer entwickeln das Produkt.
- Anforderungen technisch verstehen
- Aufwand einschätzen
- User Stories in konkrete Tasks zerlegen
- Umsetzung planen
- programmieren, testen und dokumentieren
- ein fertiges Increment liefern

### 3 Artefakte

#### Product Backlog
Das Product Backlog enthält alle bekannten Anforderungen, Ideen, Verbesserungen und Fehler des Produktes
```
Nutzer kann sich registrieren
Nutzer kann Passwort zurücksetzen
Admin kann Nutzer sperren
Zahlung über PayPal ermöglichen
```

#### Sprint Backlog
Das Sprint Backlog enthält die Arbeit, die für den aktuellen Sprint ausgewählt wurde. 
```
Product Backlog
= alles, was irgendwann relevant sein könnte

Sprint Backlog
= Arbeit für den aktuellen Sprint
```

#### Increment#
Das Increment ist das funktionierende Ergebnis eines Sprints. Das gilt ausschließlich für fertige Arbeiten, erst dann gilt ein abgeschlossenes increment.


### SCRUM Events

#### Sprint Planning
Im Sprint planning wird festgelegt
- was im Sprint umgesetzt wird
- wie die Umsetzung erfolgen soll
##### Sprint Planning 1
Was wird im kommenden Sprint umgesetzt, das für Anwender oder Kunden einen Wert besitzt?
##### Sprint planning 2
Wie wird die ausgewählte Arbeit technisch umgesetzt?

Beispiel:
User Story:
```
Als Nutzer möchte ich mein Passwort zurücksetzen können,
damit ich meinen Zugang wiederherstellen kann.
```

Tasks:
```
Reset-Endpunkt implementieren
Token generieren
Token in Datenbank speichern
E-Mail senden
Reset-Formular bauen
Tests erstellen
```


## 3. Daily Scrum

Das Daily Scrum ist eine kurze tägliche Abstimmung der Developer. Man überprüft den
- Fortschritt
- Arbeit bis zum nächsten Daily planen
- Hindernisse erkennen
- Zusammenarbeit koordinieren

## 4. Sprint Review

Im Sprint Review wird das fertige Increment gemeinsam mit Stakeholdern betrachtet.
Nur fertige Stories werden präsentiert und zählen zum Increment.

## 5. Akzeptanzkriterien

Akzeptanzkriterien beschreiben, wann eine User Story fachlich erfüllt ist.
User Story:
> Als Nutzer möchte ich mich anmelden können, damit ich auf mein Profil zugreifen kann.

Akzeptanzkriterien:
```
Login mit korrekten Daten funktioniert
Falsches Passwort zeigt verständliche Fehlermeldung
Gesperrter Nutzer kann sich nicht anmelden
Passwort wird nicht im Klartext gespeichert
```

## 6. Project Vision

Die Projekt Vision beschreibt die grobe gemeinsame Produktidee.
- Für wen bauen wir das Produkt?
- Welches Problem lösen wir?
- Welchen Nutzen bieten wir?
- Was ist die grobe Lösung?
- Was gehört ausdrücklich nicht dazu?

Kurz und ohne technische Details

## 7. Epic

Ein Epic ist ein großer fachlicher Themenblock.
Es ist:
- aus der Project Vision abgeleitet
- zu groß für einen einzelnen Sprint
- noch nicht konkret genug für die direkte Umsetzung
- eine Gruppierung mehrere zusammengehöriger Anforderungen

Man kann sich den Aufbau wie folgt vorstellen:
```
Project Vision → Epics → User Stories
```

## 8. User Story

Eine User Story beschreibt eine Anforderung aus Sicht eines Nutzers. 
```
Als "Rolle" möchte ich "Funktionen", damit "Nutzen". 
```

Eine gute User Story beschreibt:
- wer etwas möchte
- was gewünscht ist
- warum es nützlich ist


## 9. Tasks

Tasks sind konkrete technische Arbeitsschritte zur Umsetzung einer User Story.

User Story:
```
Als Spieler möchte ich meine Bounty sehen.
```

Tasks:
```
Bounty aus Datenbank laden
API-Endpunkt erstellen
Anzeige im Menü umsetzen
Fehlerfall behandeln
Tests schreiben
```


## 10. Definition of Ready

Die Definition of Ready(DoR) beschreibt, wann eine User Story ausreichend vorbereitet ist, um in einen Sprint aufgenommen zu werden. Ein Eintrag sollte nicht in den Sprint, wenn niemand versteht, was eigentlich gebaut werden soll.
Typische Kriterien:
- fachlich verständlich
- Nutzen erkennbar
- Akzeptanzkriterien vorhanden
- Abhängigkeiten bekannt
- Aufwand grundsätzlich schätzbar
- offene Fragen ausreichend geklärt

## 11. Definition of Done

Die Definition of Done(DoD) beschreibt wann Arbeit wirklich als fertig gilt.
Typische Kriterien:
- Code implementiert
- Tests erfolgreich
- Review durchgeführt
- Dokumentation aktualisiert
- Akzeptanzkriterien erfüllt
- in relevante Umgebung integriert
- keine bekannten kritischen Fehler

## 12. Sprint Review vs. Retrospektive

**Sprint Review**: Was haben wir gebaut und wie sollte sich das Produkt weiterentwickeln?
**Sprint Retrospektive**: Wie haben wir gearbeitet und was können wir im nächsten Sprint verbessern?

Sprint Backlog
