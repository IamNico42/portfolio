## 1. Transparenz

In Verteilten Systemen bedeutet Transparenz:
> Unterschiede, die durch die Verteilung entstehen, sollen für Anwender möglichst unsichtbar sein.

Man öffnet zum Beispiel eine Domain `www.no-mercy-gaming.com` und man weiß nicht
- welcher konkrete Server antwortet
- in welchem Rechenzentrum der steht
- ob die Datenbank repliziert ist
- ob ein Load Balancer davor sitzt
- ob ein Server gerade ausgefallen ist und ein anderer übernommen hat

### Warum ist Transparenz wichtig?

Weil verteilte Systeme sonst für Nutzer und Entwickler unnötig kompliziert wären.
Stell dir vor, man müsste bei jeder Datei wissen:

```
Datei liegt auf Server 12 in Frankfurt.
Kopie B liegt in München.
Server 12 ist gerade langsam.
Bitte nutze IP 123.45.67.89.
Falls Timeout, manuell Server 13 probieren.
```
Das wäre komplett unbrauchbar.
Transparenz versteckt diese Komplexität.

### Die wichtigsten Transparenzarten

##### **Zugriffstransparenz**
>Der Zugriff auf eine Ressource soll gleich aussehen, egal ob sie lokal oder entfernt ist.
>Beispiel:

```
open("datei.txt")
```

Im Idealfall merkt der Nutzer nicht, ob die Datei lokal liegt oder über ein Netzwerk-Dateisystem kommt.

##### **Benennungs-/Ortstransparenz**:
> Der Nutzer muss nicht wissen, wo eine Ressource physisch liegt.

Man verwendest einen logischen Namen statt einer konkreten Adresse.

Beispiel:
```
www.google.de
```

statt:
```
142.250.185.3
```

Oder:
```
/user/42/profile
```

statt:
```
Server-Frankfurt-03, Datenbank-Shard-7, Tabelle users, Zeile 42
```
Man benutzt den Namen. Das System findet den Ort.


##### **Migrationstransparenz**
**Migration** bedeutet:
> Eine Ressource kann ihren Ort wechseln.

Migrationstransparenz heißt:
> Vor dem Zugriff muss der Nutzer nicht wissen, dass oder wohin die Ressource verschoben wurde.

Eine Datei oder ein Dienst wird von Server A auf Server B verschoben.
Ohne Transparenz:

```
Fehler: Ressource nicht gefunden.
Bitte neue Serveradresse nutzen.
```

Mit Transparenz:
```
Eine Ressource oder Komponente kann ihren Standort zwischen Zugriffen ändern, ohne dass Nutzer ihren Zugriff anpassen müssen.
```

Migration ist stärker als reine Ortstransparenz: Die Position einer Ressource vor dem entfernten Zugriff ist nicht relevant.


##### **Relokationstransparenz**
Relokation ist nochmal stärker als Migration.

**Relokation** bedeutet:
> Eine Ressource bewegt sich während des laufenden Zugriffs.

Relokationstransparenz heißt:
> Der Nutzer merkt nicht, dass die Ressource während der Nutzung verschoben wird.

Beispiel Handy:
Man telefoniert im Auto.
Während der Fahrt wechselt dein Handy zwischen Funkzellen.

Ohne Relokationstransparenz:
```
Jedes Mal Gespräch weg.
```

Mit Relokationstransparenz:
```
Gespräch läuft weiter.
```

>[!tip] Merksatz
>Migration = Umzug vor Nutzung.
>Relokation = Umzug während Nutzung.

##### **Persistenztransparenz**
**Persistenz** bedeutet:
> Daten überleben das Ende eines Prozesses.

Transiente Daten:
```
existieren nur im Arbeitsspeicher / nur solange Prozess läuft
```

Persistente Daten:
```
bleiben gespeichert, auch wenn Prozess beendet wird
```

Persistenztransparenz heißt:
> Der Nutzer oder Entwickler muss möglichst nicht ständig unterscheiden, ob Daten gerade nur temporär oder dauerhaft gespeichert sind.

Beispiel
Man bearbeitet ein Google Docs Dokument.
Man denkt nicht ständig:
```
Ist das gerade nur im RAM?Ist es schon auf Server gespeichert?Ist es repliziert?
```
es ist einfach:
```
Mein Dokument ist da.
```

##### **Replikationstransparenz**
**Replikation** bedeutet:
> Eine Ressource existiert mehrfach als Kopie.

Warum macht man das?
- Ausfallsicherheit
- Lastverteilung
- kürzere Entfernung zum Nutzer
- bessere Performance

Ein Bild liegt vielleicht auf Servern in Frankfurt, Paris und Amsterdam.
Man ruft das Bild auf
Man weiß aber nicht:
```
Welche Kopie habe ich bekommen?
```

Problem:
Kopien müssen konsistent bleiben
##### **Fehlertransparenz**
>Fehler oder Ausfälle einzelner Komponenten sollen möglichst verborgen oder automatisch behandelt werden.

Beispiel:
Ein Server fällt aus.
Ohne Fehlertransparenz:
```
Website kaputt.
```

**Mit Fehlertransparenz:**
```
Load Balancer schickt dich zu anderem Server.Du merkst nichts.
```

Oder:
Eine Nachricht geht verloren.
Das System versucht es nochmal.
```
Request failed → retry → success
```

Fehlertransparenz heißt nicht, dass Fehler unmöglich sind.
Es heißt nur:
```
Das System versucht, sie vor dem Nutzer zu verstecken oder automatisch zu behandeln.
```

>[!tip] **Transparenz ist nicht perfekt erreichbar**
>Verteilung kann nie vollständig transparent sein. Gründe sind unter anderem Latenz, Kommunikationszeit und unvorhersehbare Ausnahmen. Eine Komponente kann nicht antworten; oft kann man nicht sicher unterscheiden, ob sie langsam oder ausgefallen ist. Maßnahmen sind Timer und mehrere Kommunikationsversuche, aber das kann Sekunden dauern.

>[!tip] **Transparenz kann auch gefährlich sein**
> Zu viel Transparenz kann Nutzer oder Entwickler täuschen
> Wenn ein Remote-Aufruf genauso ausseht wie ein lokaler Funktionsaufruf, vergisst man leicht: 
> - Der Aufruf kann langsam sein.
 >- Der Aufruf kann fehlschlagen.
 >- Der Aufruf kann Netzwerkfehler haben.

## 2. Kohärenz: Das System wirkt wie eine Einheit

Kohärenz bedeutet:

> Ein verteiltes System ist eine Sammlung unabhängiger Computer, die für Nutzer wie ein einzelnes zusammenhängendes System erscheint.

bzw.
- zusammenhängend
- einheitlich
- stimmig
- nicht wie viele getrennte Einzelteile

Beispiel Google Drive:
Technisch liegen Dateien verteilt auf Servern, repliziert, gesichert, vielleicht in verschiedenen Rechenzentren. Aber es wirkt wie "Mein Drive".
Das System wirkt kohärent.

**Qualitätsmerkmal** und **Entwurfskriterium**


> [!tip] Merksätze
Transparenz = Verteilung möglichst unsichtbar machen.
Kohärenz = viele unabhängige Rechner wirken wie ein System.
Ortstransparenz = Name statt physischer Adresse.
Migration = Ressource zieht vor Zugriff um.
Relokation = Ressource zieht während Zugriff um.
Replikation = mehrere Kopien, Nutzer merkt es nicht.
Fehlertransparenz = Ausfälle möglichst verstecken.
Vollständige Transparenz gibt es nicht, weil Netzwerk nie perfekt ist.
