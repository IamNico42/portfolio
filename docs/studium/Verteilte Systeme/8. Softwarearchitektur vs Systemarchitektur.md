## 1. Softwarearchitektur

Die Softwarearchitektur beschreibt den logischen Aufbau der Software - also die Struktur der Software welche sich folgende Fragen stellt:
- Welche Komponenten gibt es?
- Welche Aufgaben haben diese?
- Wer kommuniziert mit welcher Komponente?
- Welche Schnittstellen existieren

```
Browser
↓
Frontend
↓
Backend
↓
Payment Service
↓
Datenbank
```

## 2. Systemarchitektur

Bei der Systemarchitektur geht es um **Rechner, Server, Netzwerke, VM's, Container, Cloud, Standorte**. Also um den Physischen Faktor:
```
[Server 1]
Frontend
Backend

------------
[Server 2]
Datenbank

------------
[Server 3]
Payment Service
```
Systemarchitektur beschreibt die Verteilung der Software auf reale Systeme.


## 3. Vergleich und warum braucht man beides

Die Softwarearchitektur beschreibt den logischen Aufbau eines Softwaresystems. Sie legt fest, welche Komponenten existieren, welche Aufgaben sie übernehmen und wie sie miteinander kommunizieren. Die Systemarchitektur beschreibt dagegen den physischen Aufbau eines Systems. Sie legt fest, auf welchen Rechnern, Servern oder Netzwerken die Softwarekomponenten tatsächlich ausgeführt werden. Eine Softwarearchitektur kann auf unterschiedlichen Systemarchitekturen betrieben werden.

>[!tip] Merksätze
> Softwarearchitektur := logischer Aufbau
> Systemarchitektur := physischer Aufbau
> 
> Software beantwortet:
> "Was gibt es?"
> 
> System beantwortet:
> "Wo läuft es?"
> 
> Eine Softwarearchitektur kann auf vielen verschiedenen Systemarchitekturen laufen.

## 4. Architekturstile

Ein **Architekturstil** beschreibt **ein allgemeines Entwurfsmuster**, nach dem ein System aufgebaut wird.

#### Schichtenarchitektur
Man teilt die Software auf mehrere Schichten auf:
```
Präsentation(GUI)
──────────────
Anwendungslogik(Berechnungen, Login check)
──────────────
Datenzugriff(SELECT usw.)
──────────────
Datenhaltung(Datenbank)
```


#### Objektorientierte Architektur
Das System wird als Menge von Objekten oder Komponenten betrachtet.
Jedes Objekt besitzt:
- Einen Zustand
- bietet Methoden an
- kommuniziert über Methodenaufrufe.
```
kunde.bestellungAufgeben()
```
Verteilt kann daraus ein entfernter Methodenaufruf, etwa RMI, oder ein Service-Aufruf werden.


#### Datenorientierte Architektur
Hier stehen gemeinsame Daten im Mittelpunkt. Mehrere Komponenten greifen auf einen gemeinsamen Datenbestand zu.

```
Komponente A ─┐
Komponente B ─┼──> gemeinsame Datenbank
Komponente C ─┘
```

**Vorteil**: Einheitliche Daten
**Probleme**: Flaschenhals, Ausfall der Datenquelle, Nebenläufigkeit und Konsistenz

#### Ereignisorientierte Architektur
Komponenten kommunizieren über Ereignisse.

```
Bestellung erstellt
→ Zahlungsdienst reagiert
→ Lagerdienst reagiert
→ Versanddienst reagiert
```

Typische Begriffe:
- Event
- Producer
- Consumer
- Event Bus
- Message Broker

**Vorteil:** lose Kopplung und gute Erweiterbarkeit.  
**Nachteil:** Abläufe sind schwieriger nachzuverfolgen und Fehlerbehandlung wird komplexer.


#### Client-Server Architektur

Der Client nutzt einen Dienst und initiiert die Kommunikation. Der Server bietet den Dienst an und wartet auf Anfragen.

**Vorteile:** klare Rollen und zentrale Dienstbereitstellung.  
**Nachteile:** Server kann Flaschenhals oder Single Point of Failure sein.

#### Event-Driven Architecture, EDA

Das ist die konkrete verteilte Ausprägung einer ereignisorientierten Architektur. EDA ist dabei häufig asynchron. Die beteiligten Dienste müssen nicht gleichzeitig verfügbar sein, wenn ein Broker die Ereignisse zwischenspeichert.

```
Producer
   ↓ Event
Message Broker
   ↓
Consumer A
Consumer B
Consumer C
```

```
Event: Zahlung erfolgreich
→ Bestellservice markiert Bestellung bezahlt
→ Versandservice bereitet Versand vor
→ E-Mail-Service versendet Bestätigung
```


#### Mehrschichtarchitektur

Das ist die Weiterentwicklung der Schichtenidee auf mehrere Systemeinheiten.

**2-Tier**
Client <-> Datenbankserver

**3-Tier**
Browser -> Backend -> Datenbank

**n-Tier**
Weitere Ebenen kommen hinzu:
```
Frontend
-> API-Gateway
-> Anwendungsservices
-> Cache
-> Datenbanken
```

#### Serviceorientierte beziehungsweise komponentenbasierte Architektur

Das System besteht aus fachlich getrennten Diensten oder Komponenten:
```
User Service
Payment Service
Inventory Service
Notification Service
```

Jeder Dienst bietet eine definierte Schnittstelle und kann intern unterschiedlich umgesetzt sein. Das kann später zu Microservices führen, wobei „Microservices“ nur eine spezielle Form davon sind.



### Verhältnis der Architekturen

Diese Architekturen schließen sich nicht gegenseitig aus.

Ein System kann gleichzeitig:
- intern eine **Schichtenarchitektur** haben,
- nach außen **Client-Server** verwenden,
- Daten in einer **datenorientierten Architektur** speichern,
- und zwischen Diensten Ereignisse über **EDA** austauschen.

```
Browser → Webshop-Backend → Datenbank
                  ↓
           Event: Bestellung erstellt
                  ↓
       Payment / Versand / E-Mail
```

Das Backend kann intern geschichtet sein, Browser und Backend bilden Client-Server, die Datenbank ist datenorientiert und die weiteren Dienste reagieren ereignisorientiert.


### Thin Client
Ein Thin Client enthält nur wenig eigene Anwendungslogik zB: Eine Browseranwendung (Word im Web)
```
Client:
- Darstellung
- Input

Server:
- Datenhaltung
- Datenverarbeitung
- Logik
```


### Fat Client
Ein Fat Client hat viel Anwendungslogik: Eine Desktopawendung (Word auf Desktop lokal)
```
Client:
- Darstellung
- Logik
- teilweise lokale Datenverarbeitung
  

Server:
- Datenhaltung oder einzelne Dienste
```
