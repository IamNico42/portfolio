Es gibt verschiedene Möglichkeiten. Manche sind sehr technisch, andere verstecken fast die komplette Netzwerkkommunikation.
Man spricht deshalb vom **Kommunikationsgrad** bzw. **Abstraktionsgrad der Kommunikation**.
Je höher der Abstraktionsgrad, desto weniger muss sich der Entwickler um Netzwerkdetails kümmern.

## 1. Socket

Der **Socket** ist die **unterste Kommunikationsebene**. Ein Socket ist nichts anderes als ein **Kommunikationsendpunkt**. Man kann sich einen Socket wie einen Telefonanschluss vorstellen. Er verbindet zwei Programme.

## 2. Nachrichtenkommunikation

Man verschickt nicht einfach nur rohe Bytes. Man verschickt **Nachrichten**. zB als **JSON** verpackt.

## 3. RPC (Remote Procedure Call)

RPC bedeutet: **Ein entfernter Funktionsaufruf.** Man ruft eine Funktion auf welche eigentlich auf einem anderen Rechner läuft. RPC versteckt in diesem Fall die Verteilung weil der Aufruf wie ein lokaler Aufruf aussieht. Dadurch kann es aber auch langsam sein, fehlschlagen und für timeouts sorgen und ist somit fehleranfälliger.
### **Marshalling**
Beim ausführen von verteilten Funktionen mit **Objekten** im Parameter haben wir das Problem, dass wir das Objekt nicht als Nachricht über das Netzwerk versenden können. Dafür gibt es dann **Marshalling und Unmarshalling**
```
Marshalling
=
Objekte -> Nachrichten

Unmarshalling
=
Nachrichten -> Objekte
```
### **Stub (Client-Stub, Server-Stub)**

Der Stub ist ein Stellvertreter - also eine Hilfsklasse, Bibliothek oder automatisch erzeugte Proxy-Klasse.
Der Stub übernimmt:
- Marshalling
- Netzwerkkommunikation
- Fehlerbehandlung
Für das Programm sieht alles lokal aus.
## 4. RMI (Remote Method Invocation)

Das ist prinzipiell das gleiche wie RPC nur für Methoden.
```
RPC:berechnePreis()
RMI:produkt.berechnePreis()
```

**RPC**
-> Funktionsaufruf

**RMI**
-> Methodenaufruf auf Objekten
## Vorteile des höheren Kommunikationsgrades

Je höher der Kommunikationsgrad, desto einfacher wird das Programmieren.
Man muss sich weniger Gedanken machen über:
- Netzwerk
- Bytes
- Protokolle
- Marshalling

### Nachteile
Je höher, desto weniger Kontrolle.
Außerdem entstehen:
- zusätzlicher Overhead
- zusätzliche Bibliotheken
- mehr Abstraktion
- manchmal schlechtere Performance

## Abschluss

>[!tip] Merksätze
> Socket = Kommunikation auf Byte-Ebene.
> Nachrichten = strukturierte Informationen.
> RPC = entfernter Funktionsaufruf.
> RMI = entfernter Methodenaufruf auf Objekten.
> Marshalling = Objekte in Nachrichten umwandeln.
> Unmarshalling = Nachrichten wieder zu Objekten.
> Stub = Stellvertreter des entfernten Dienstes.
> 
> Je höher der Kommunikationsgrad, desto einfacher wird die Programmierung,
> aber desto weniger Kontrolle hat der Entwickler.
