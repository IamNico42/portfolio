    2. Lokal vs Verteilt

    1. Der große Unterschied: lokal vs. verteilt¶

    Lokal¶

    In einem normalen lokalen Programm machst du sowas:

    user = database.getUser(42);

    Das fühlt sich direkt an:

    Das fühlt sich direkt an:

    Verteilt¶

    In einem verteilten System geht das nicht direkt, weil die andere Komponente auf einem anderen Rechner läuft.

    Dann wird daraus: Komponente A sendet Nachricht → Netzwerk → Komponente B verarbeitet → Antwort zurück

    Zusammengefasst: 

    Bei lokaler Kommunikation befinden sich aufrufende und aufgerufene Teile auf demselben Rechnersystem. Bei verteilter Kommunikation muss der Informationsaustausch zwischen verschiedenen Rechnersystemen über ein Netzwerk erfolgen.

    2. Kommunikation¶

    Kommunikation heißt erstmal nur:

    Eine Komponente überträgt Information an eine andere Komponente.

    Kommunikation kann sein:

    • Anfrage
    • Antwort
    • Event
    • Nachricht
    • HTTP-Request
    • API-Call
    • Datenpaket

    Kommunikation ist der Austausch von Nachrichten zwischen Komponenten eines verteilten Systems.

    3. Interaktion¶

    Interaktion ist Kommunikation mit Beziehung zwischen den Beteiligten.

    A tut etwas → B reagiert darauf

    z. B. Request-Response-Modell 

    Client sendet Request
Server verarbeitet Request
Server sendet Response
Client verarbeitet Response

    Also: 

    Kommunikation = Nachricht wird ausgetauscht
Interaktion = Nachricht löst Reaktion aus

    4. Kooperation¶

    Kooperation geht noch eine Stufe weiter.

    Kooperation heißt:

    Mehrere Komponenten arbeiten gemeinsam an einem Ziel.

    Beispiel Webshop mit PayPal: User-Browser → Shop-Backend → PayPal → Shop-Backend → Datenbank → Browser Ziel: Bestellung erfolgreich abschließen.

    Dafür müssen mehrere Komponenten zusammenarbeiten:

    • Browser zeigt Bestellung an
    • Shop-Backend verarbeitet Warenkorb
    • PayPal wickelt Zahlung ab
    • Datenbank speichert Bestellung
    • vielleicht Versanddienst bekommt Auftrag

    Das ist mehr als nur eine Nachricht. Es ist ein gemeinsamer Ablauf.

    Die drei Begriffe sauber auseinanderhalten¶

    Begriff Bedeutung Beispiel
    Kommunikation Nachrichten werden ausgetauscht Client sendet HTTP-Request
    Interaktion Komponenten reagieren aufeinander Server beantwortet Request
    Kooperation Komponenten arbeiten gemeinsam an Ziel Shop + PayPal + DB führen Bestellung aus

    5. Request-Response-Modell¶

    Das ist der Klassiker. Ablauf: 

    1. Server wartet2. Client sendet Anfrage3. Server empfängt Anfrage4. Server verarbeitet Anfrage5. Server sendet Antwort6. Client verarbeitet Antwort

    Beispiel Login: 

    Client: POST /login mit Username/PasswortServer: prüft DatenServer: sendet "Login erfolgreich" oder "401 Unauthorized"

    6. Synchron¶

    Der Client wartet auf die Antwort, bevor er weitermacht.

    Beispiel Login: 

    Du klickst "Einloggen"App wartetServer prüftApp bekommt AntwortDann erst geht es weiter

    Das ist sinnvoll, wenn du sofort wissen musst, ob etwas funktioniert hat. Typische synchrone Fälle: - Login - Datei abrufen - Produktdetails laden - Preis berechnen - Zahlung bestätigen

    Vorteil: 

    Einfach zu verstehen, klare Antwort.

    Nachteil: 

    Client blockiert/wartet.Wenn Server langsam oder nicht erreichbar ist, hängt der Ablauf.

    Bei synchroner Kommunikation wartet der Sender beziehungsweise Client nach einer Anfrage auf die Antwort, bevor er die Verarbeitung fortsetzt.

    7. Asynchron¶

    Der Client sendet etwas ab und wartet nicht blockierend auf die direkte Antwort.

    Beispiel:

    Video hochladenServer sagt: "Upload erhalten, Verarbeitung läuft"App kann weiter benutzt werdenSpäter kommt: "Video ist fertig verarbeitet"

    Typische asynchrone Fälle: - Videoverarbeitung - E-Mail-Versand - Benachrichtigungen - Hintergrundjobs - lange Berechnungen - Bestellung wird später bestätigt

    Vorteil: 

    Client blockiert nicht.System bleibt reaktiver.

    Nachteil: 

    Komplexer, weil Ergebnis später kommt.Man braucht Status, Benachrichtigung, Fehlerbehandlung.

    Bei asynchroner Kommunikation wartet der Sender nach dem Absenden einer Nachricht nicht blockierend auf die Antwort. Das Ergebnis wird später verarbeitet oder über eine separate Benachrichtigung bereitgestellt.

    Asymmetrie¶

    Client und Server haben unterschiedliche Rollen.

    Client fragt an.
    Server wartet und antwortet.

    Das ist Rollenverteilung.

    Asymmetrie = Wer macht was?

    Beispiel: Ein klassischer Webserver ist meistens asymmetrisch, weil Client und Server unterschiedliche Rollen haben.

    8. Multi-Request und Multi-Client¶

    Ein echter Server bedient normalerweise nicht nur einen Nutzer.

    Client A ─┐
Client B ─┼──> Server
Client C ─┘

    Der Server muss mehrere Anfragen handhaben.

    Das führt später direkt zu Nebenläufigkeit:

    Mehrere Clients greifen gleichzeitig auf denselben Dienst/Ressource zu.

    Beispiel:

    100 Nutzer öffnen gleichzeitig deine Website.
20 Spieler senden gleichzeitig Bewegungen an den Gameserver.
2 Nutzer ändern gleichzeitig denselben Datenbankeintrag.

    Dann muss das System koordinieren.

    9. Dienstkomposition: Server wird selbst zum Client¶

    Beispiel: 

    Browser → Shop-Backend → PayPal

    Für den Browser ist dein Shop-Backend der Server. Aber gegenüber PayPal ist dein Shop-Backend plötzlich Client.

    Also eine Komponente kann gleichzeitig Server und Client sein - je nachdem, in welcher Beziehung man sie betrachtet.

    Ein Server kann zur Beantwortung einer Anfrage Dienste anderer Server benötigen und tritt dann selbst als Client für einen weiteren Server auf. Dadurch entstehen komplexere Dienste durch Zusammensetzung anderer Dienste. Nachteile sind erhöhter Kommunikationsaufwand und Synchronisierungsaufwand bei gleichzeitiger Subdienst-Ausführung.

    Beispiel: 

    Client fragt Server A: "Bestellung abschließen"Server A fragt Server B: "Zahlung durchführen"Server B antwortet Server AServer A antwortet Client

    Diagramm: 

    Client → Server A → Server BClient ← Server A ← Server B

    Das ist extrem wichtig:

    Rollen sind relativ zur jeweiligen Kommunikation.

    10. Zusammenfassung¶

    Kommunikation = Komponenten tauschen Nachrichten aus.
Interaktion = Komponenten reagieren aufeinander.
Kooperation = Komponenten arbeiten gemeinsam an einem Ziel.
Synchron = Client/Sender wartet auf Antwort.
Asynchron = Client/Sender wartet nicht blockierend.
Asymmetrie = Client und Server haben unterschiedliche Rollen.
Ein Server kann gleichzeitig Client eines anderen Servers sein.
