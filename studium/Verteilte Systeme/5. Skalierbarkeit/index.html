    5. Skalierbarkeit

    1. Was bedeutet Skalierbarkeit¶

    Ein System kann größer werden, ohne dass man de Grundidee, Architektur, Protokolle oder Algorithmen komplett ändern muss

    Ein System ist also skalierbar, wenn es mit Wachstum umgehen kann - kann bedeuten: 

    mehr Nutzer
mehr Anfragen
mehr Daten
mehr Server
mehr Standorte
mehr Organisationen

    Vertikale Skalierung:¶

    1 Server wird stärker: - mehr CPU - mehr RAM - mehr Speicher

    Horizontale Skalierung:¶

    1 Server → 2 Server → 10 Server → 100 Server

    2. Drei Dimensionen von Skalierbarkeit¶

    Größenskalierbarkeit:¶

    Das System kann mit mehr Teilnehmern, Komponenten, Nutzern, Daten oder Anfragen umgehen.

    Beispiel: 

    100 Nutzer → 10.000 Nutzer → 1.000.000 Nutzer
    Oder: 
    1 Service → 20 Services
    Oder: 
    10 GB Daten → 10 TB Daten

    Problem¶

    Je mehr Nutzer und Komponenten, desto mehr: 

    Anfragen
Datenverkehr
gleichzeitige Zugriffe
Datenbanklast
Koordinationsaufwand
Fehlerquellen

    Beispiel Webshop¶

    Am Anfang: 

    1 Webserver
1 Datenbank
    Später: 
    5 Webserver
1 Load Balancer
mehrere Datenbank-Replikate
CacheQueuePayment-Service
Search-Service
    Das System wird größer, muss aber trotzdem funktionieren.

    Typische Lösungen¶
    • Load Balancing
    • Caching
    • Replikation
    • Datenbank-Sharding
    • Queues
    • Microservices
    • stateless Server
    Geographische Skalierbarkeit:¶

    Das System kann über größere räumliche Entfernungen funktionieren.

    Also nicht nur: 

    alle Server in einem Raum
    sondern: 
    Server in DeutschlandServer in USAServer in AsienNutzer weltweit

    Bei geographischer Skalierung gelten besonders Herausforderungen wie Latenz, zusätzliche Hops, lange Antwortzeiten und Flaschenhälse.

    Was ist Latenz?¶

    Latenz ist die Verzögerung, bis eine Nachricht ankommt und eine Antwort zurückkommt.

    Was sind Hops?¶

    Ein Hop ist ein Zwischenschritt im Netzwerk. 

    Client → Router → Provider → Backbone → Rechenzentrum → Server
    Je mehr Zwischenstationen, desto mehr potenzielle Verzögerung oder Fehlerstelle.

    Daten näher an Nutzer bringen
    Zum Beispiel durch: - CDN - regionale Server - Edge-Server - Replikation - Caching

    Administrative Skalierbarkeit¶

    Das System funktioniert auch, wenn mehrere Organisationen, Rechtebereiche oder Verwaltungsdomänen beteiligt sind.

    Beispiel: 

    Dein Webshop
PayPal
Versanddienst
Steuerdienst
Bank
externe Login-Anbieter
    Diese System gehören einem nicht, aber müssen trotzdem zusammenarbeiten.

    Verschiedene Organisationen haben unterschiedliche: 

    Sicherheitsregeln
Datenformate
APIs
Verfügbarkeiten
Verantwortlichkeiten
Datenschutzvorgaben
Zugriffsrechte

    Eine Hochschule hat vielleicht: 

    Moodle
LSF
Prüfungsverwaltung
E-Mail
Bibliothekssystem
Microsoft/Teams
Shibboleth/Login

    Alles wird von unterschiedlichen Stellen oder Systemen verwaltet. Trotzdem soll es für Studierende möglichst zusammenhängend funktionieren.

    3. Warum ist Skalierbarkeit schwierig?¶

    Weil Wachstum nicht nur "mehr vom Gleichen" ist. Sondern man muss die gesamte Infrastruktur anpassen und so abstrakt wie möglich gestalten.

    Mit wachsender Zahl von Nutzern und Komponenten steigt die Last; abhängig von Architektur und Kommunikationsstruktur kann sie auch überproportional wachsen. 

    mehr Kommunikation
mehr Koordination
mehr gleichzeitige Zugriffe
mehr Datenkonsistenzprobleme
mehr Fehlerquellen
mehr Sicherheitsfragen

    4. Typische Skalierungsprobleme¶

    4.1 Zentraler Flaschenhals¶

    Wenn: 

    Client A ┐
Client B ├──> zentraler Server → Datenbank
Client C ┘

    Dann wird dieser zentrale Server irgendwann überlastet sein

    Lösung: - mehrere Server - Load Balancer - Caching - Replikation

    4.2 Zu viel Kommunikation¶

    Wenn jede Komponente mit jeder anderen reden muss dann explodiert der Kommunikatonsaufwannd.

    4.3 Zu viel Synchronisation¶

    Wenn alle ständig denselben Zustand abgleichen müssen, wird es langsam. Overload durch Updates.

    4.4 Latenz¶

    Geographische Entfernungen erhöht die Antwortzeiten.

    4.5 Teilausfälle¶

    Je mehr Komponenten, desto wahrscheinlicher ist, dass irgendeine gerade nicht erreichbar ist.

    5. Strategien für bessere Skalierbarkeit¶

    5.1 Caching¶

    Daten werden zwschengespeichert.

    5.2 Replikation¶

    Daten oder Dienste werden mehrfach bereitgestellt.

    5.3 Load Balancing¶

    Ein Load Balancer verteilt Anfragen auf mehrere Server und damit wird die Last von vornherein verteilt.

    5.4 Sharding¶

    Daten werden aufgeteilt. Beispiel: 

    User A-M auf Datenbank 1User N-Z auf Datenbank 2
    Oder: 
    Europa-Daten auf EU-DBUSA-Daten auf US-DB

    Asynchrone Verarbeitung¶

    Nicht alles muss sofort passieren.

    Beispiel: Bestellung eingehen -> Sofort bestätigen Rechnung erzeugen -> Im Hintergrund etwas verzögert E-mail senden -> Queue

    Vorteil: 

    Nutzer wartet wenigerSystem kann Last abfedern
    Nachteil: 
    Status und Fehlerbehandlung werden komplexer

    6. Skalierbarkeit vs. Transparenz¶

    Skalierbarkeit und Transparenz hängen zusammen. Beispiel: Man fügt 10 Server hinzu Der Nutzer soll aber nicht merken:

    Heute antwortet Server 1.
Morgen antwortet Server 7.
Bild kommt von CDN.
Daten kommen aus Replikat B.

    [!tip] Merksätze - Skalierbarkeit = wachsen können, ohne komplett umzubauen. - Größe = mehr Nutzer/Komponenten/Daten. - Geographie = größere Entfernung, mehr Latenz/Hops. - Administration = mehrere Organisationen/Verantwortlichkeiten. - Mehr Skalierung = mehr Kommunikation, mehr Koordination, mehr Fehlerquellen. - Caching macht schneller, kann aber veraltet sein. - Replikation verbessert Verfügbarkeit/Performance, erschwert Konsistenz. - Load Balancing verteilt Last. - Sharding teilt Daten auf.
